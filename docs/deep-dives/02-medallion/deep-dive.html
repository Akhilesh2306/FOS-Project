Medallion Flow

FOS uses a four-tier Medallion Architecture implemented entirely inside Snowflake. Every transformation from Silver onward is a Snowflake Dynamic Table. There is no dbt, no Airflow, no external orchestrator. Snowflake itself manages the DAG of dependencies and refreshes downstream tables automatically when upstream data changes.
Bronze (35) FOS_PROD.BRONZE
Raw data, no transforms
→
Silver (35) FOS_PROD.SILVER
Cleaned, typed, enriched
→
Gold (27) FOS_PROD.GOLD
Business logic, scoring, joins
→
Semantic (11) FOS_PROD.SEMANTIC
Agent-facing SQL views
Auto-refresh cascade: OpenFlow lands data in Bronze every 15 min. Snowflake detects the change and refreshes Silver, then Gold, automatically. Semantic views are always current on query.
Stage Schema Object Type Count Refresh Mechanism
Bronze FOS_PROD.BRONZE Regular tables + Hybrid Tables 35 OpenFlow (SFDC), connector TBD (Gong), direct write (agents)
Silver FOS_PROD.SILVER Dynamic Tables 35 Auto-refresh when Bronze changes
Gold FOS_PROD.GOLD Dynamic Tables 27 Auto-refresh when Silver changes
Semantic FOS_PROD.SEMANTIC SQL Views 11 Always current on query (no materialization)
Bronze Sources
9 Salesforce tables ingested via OpenFlow every 15 min.
16 Gong tables with connector TBD.
10 FOS Hybrid Tables written directly by agents via Snowpark SDK.
Silver Transforms
1:1 mapping with Bronze. Standard cleaning: snake_case naming, type casting, null defaults, soft-delete filtering, deduplication, UTC normalization, and single-entity computed fields.
Gold Business Logic
Cross-entity joins, scoring formulas, and aggregations organized into 6 business domains. Health scoring, engagement analytics, Gong intelligence, pipeline forecasting, AI agent ops, and content management.

Why Dynamic Tables

All Silver and Gold transformations are Snowflake Dynamic Tables. We are not using dbt, Airflow, Dagster, or any external transformation orchestrator. This decision trades away dbt's mature testing ecosystem in exchange for zero-infrastructure orchestration and near-real-time refresh.
Factor Dynamic Tables dbt
Orchestration Snowflake manages the DAG natively Requires external scheduler (Airflow, cron, dbt Cloud)
Refresh trigger Automatic on upstream data change Scheduled or manually triggered
Latency Near-real-time (configurable target lag) Batch (typically hourly or longer)
Infrastructure Zero additional infrastructure Needs dbt Core/Cloud + scheduler + CI/CD pipeline
Cost model Snowflake compute credits when refresh runs Snowflake compute + orchestrator hosting costs
Learning curve SQL only, no Jinja templating SQL + Jinja + YAML config + dbt-specific concepts
Lineage Built into Snowflake UI and metadata Requires dbt docs or external tooling
Testing Snowflake data quality checks (limited) dbt tests are mature and extensible
What We Gain
Zero external orchestration infrastructure. No Airflow, no cron, no CI/CD pipeline for transforms.

Automatic DAG management. Snowflake infers dependencies from SQL definitions.

Near-real-time refresh without polling or scheduling.

Everything stays inside the Snowflake security perimeter. Data does not leave Snowflake for transformation.

Simpler operational footprint. One fewer system to monitor and maintain.
What We Lose
dbt's mature testing framework (schema tests, data tests, custom generic tests).

Incremental model strategies (merge, delete+insert, insert_overwrite).

Version-controlled transformations in Git with PR review. Dynamic Tables are defined in Snowflake, not in repo files.

Community packages (dbt_utils, codegen, audit_helper).

Easy dry-run / compile-only mode for development.
Cost at MVP scale: Dynamic Tables consume Snowflake credits only when they refresh. For FOS at MVP scale (tens of opportunities, not millions), costs should be negligible. At production scale with full pipeline volume, credit consumption needs monitoring and target lag tuning.

Write-Back Loop

Agents read from Semantic views, generate outputs, and write those outputs back to Bronze Hybrid Tables. Those Bronze tables feed into Silver and Gold Dynamic Tables. The outputs become inputs, making agent results first-class citizens in the data model.
Agent Reads Semantic view
→
Generates NBA / Playbook / Score
→
Writes Hybrid Table Bronze (e.g., fos_next_best_actions)
→
Auto-refresh Silver Dynamic Table
→
Auto-refresh Gold Dynamic Table
→
Semantic Reflects new data on next query
Agent outputs flow through the same pipeline as Salesforce and Gong data, enabling cross-entity analytics.
NOT an infinite loop. Agents are event-driven, not continuous. The Detection Agent runs on a 60-minute Snowflake Task schedule. NBA and Playbook agents are invoked by explicit triggers (events from Detection Agent or user requests via Slack/Web). They do not poll for changes. Dynamic Table refresh is idempotent. The data path is acyclic in practice because no part of the pipeline triggers the agent to run again.
Why This Is Intentional
Agent outputs are first-class citizens in the data model. When a dashboard queries "active recommendations for this deal," the data comes through the same pipeline as everything else. There is no separate API-only data silo. Agent outputs sit alongside Salesforce and Gong data, enabling analytics like "deals where agent recommendations were followed had 15% higher win rates."
Potential Concerns
Stale reads during refresh: If an agent queries Semantic while a Gold refresh is in progress, it reads the previously materialized data. The data is at most minutes old. Acceptable for FOS.

Write amplification at scale: If many agents write to many Hybrid Tables simultaneously, the cascade of Silver and Gold refreshes could consume significant compute. This needs monitoring at production scale.

Target Lag

Each Dynamic Table has a target lag setting that controls maximum staleness. This is a guarantee, not a fixed schedule. If upstream data has not changed, Snowflake does not refresh. If upstream data changes, Snowflake refreshes within the target lag window.
Tier Target Lag Rationale
Silver 5 minutes downstream of Bronze Fast enough for near-real-time. Affordable at MVP scale.
Gold (most views) 10 minutes downstream of Silver Business logic views do not need sub-minute freshness.
Gold (v_opportunity_llm_context) 5 minutes downstream of Silver Agents need this view to be as fresh as practical. Most critical view in the system.
Failure Handling
If a Dynamic Table refresh fails, Snowflake retries automatically. The previous materialized result remains available for reads. Queries do not fail. The table enters a "refresh failed" state visible in INFORMATION_SCHEMA and the Dynamic Table monitor.
Monitoring
Track refresh frequency, actual lag vs. target, failures, and credit consumption across all 62 Dynamic Tables (35 Silver + 27 Gold) via INFORMATION_SCHEMA.DYNAMIC_TABLE_REFRESH_HISTORY and DYNAMIC_TABLE_GRAPH_HISTORY.
Cost Drivers
Frequency of upstream changes. SFDC data arrives every 15 min via OpenFlow.
Complexity of transformation. Simple type-cast costs almost nothing. Multi-join scoring views cost more.
Target lag setting. Shorter lag means more frequent refresh checks.

Data Quality

Validation responsibility increases at each tier. Bronze stores data exactly as received with no validation. Silver enforces structural consistency. Gold applies business rules and surfaces quality problems. The v_data_completeness_score Gold view is the primary quality signal consumed by agents.
Tier Validation Level What Gets Checked
Bronze None Data stored exactly as it arrives from source. If Salesforce sends a NULL close_date, Bronze stores a NULL. If Gong sends a duplicate, Bronze stores the duplicate. Only confirmation is that ingestion succeeded (row counts, schema match).
Silver Structural Type casting (strings that should be dates). Null defaults (0 for amounts, FALSE for booleans). Deduplication from connector resyncs. Soft-delete filtering. Range validation (negative amounts, future dates).
Gold Business Rules Health score bounds (must be 0-100). MEDDPICC completeness tracking. Staleness thresholds by stage. Data completeness scoring (0-100). Referential integrity (opportunities without accounts, contacts without valid account IDs).
Agent confidence modulation: Agents use the v_data_completeness_score to modulate confidence. If the quality score is below a threshold, the agent tells the user: "I don't have enough data to recommend confidently. Here is what is missing."
Gaps in current design: There is no automated data quality testing framework (like dbt tests or Great Expectations). There is no automated alerting when quality degrades. Schema drift detection (source system adds or removes columns) is not handled.

Decisions

Key architectural decisions that have been made and open questions that still need resolution. Decided items are locked. Open items need research and team input before they can be resolved.
Decided
  • Dynamic Tables for all Silver and Gold transformations. No dbt, no Airflow, no external orchestrator.
  • AHP health score weights: Activity 30%, Threading 25%, Velocity 20%, MEDDPICC 15%, Close Date 10%.
  • Hybrid Tables for write-back. Agents write to Bronze Hybrid Tables via Snowpark SDK. OLTP-grade ACID writes, no CDC or intermediate store needed.
  • Agents query Semantic only. Agents never touch Bronze, Silver, or Gold directly. The 11 Semantic views are the sole interface for AI consumers.
  • Salesforce ingestion via OpenFlow with 15-minute incremental sync.
  • MEDDPICC scoring: 8 components (Metrics, Economic Buyer, Decision Criteria, Decision Process, Identify Pain, Champion, Competition, Paper Process), each scored 0-3, total 24.
  • Staleness thresholds are stage-dependent: Qualification 14 days, Evaluation 7 days, Negotiation 3 days.
  • Everything runs inside Snowflake's security perimeter. Data does not leave Snowflake for transformation.
Open
  • Dynamic Tables from Hybrid Tables refresh triggers. Hybrid Tables are OLTP. Dynamic Tables typically source from OLAP tables. Need to confirm refresh triggers work correctly with Hybrid Table upstream sources.
  • Credit consumption at production scale. MVP has tens of opportunities. Full pipeline has thousands. Will the cascade of 35 Silver + 27 Gold refreshes every 15 minutes be cost-effective?
  • Schema drift handling. What happens when Salesforce admins add or rename fields? Silver Dynamic Tables may break if they reference hardcoded column names.
  • Historical backfill strategy. Dynamic Tables are designed for forward-looking refresh. How do we handle the initial load of historical data?
  • Gong connector. Without this, 16 Bronze tables have no data source. Domain 3 (Gong Intelligence) is blocked. Fivetran is the fallback.
  • No data quality framework. No automated testing (like dbt tests or Great Expectations), no automated alerting, no schema drift detection. This is a known gap.
