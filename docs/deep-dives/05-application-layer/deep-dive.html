Application Layer

Frontend, backend, API, pub/sub, and auth. How users reach the agents.

1

Component Topology

The application layer is the connective tissue between the people who use FOS (Account Executives, Sales Engineers, Managers) and the AI agents that run inside Snowpark Container Services. Two delivery channels (Slack and a Web Dashboard) talk to a single FastAPI backend. Asynchronous communication is handled by Azure Event Hubs.
Component Where It Runs Why There
Web Dashboard Azure (Static Web Apps or Container Apps) Needs public URL for browser access
Slack Bot Azure Container Apps Needs public URL for Slack webhook delivery
FastAPI + Agents SPCS (Snowflake) Co-located with data, Cortex LLM, Hybrid Tables
Event Hubs Azure Snowflake has no native real-time pub/sub service
Streamlit Snowflake (Streamlit in Snowflake) Quick dev-only access, test harness only
Decided: All AI inference stays inside Snowflake. When FastAPI routes a request to an agent, the agent queries the Semantic layer and calls Cortex LLM. Everything happens inside the Snowflake security perimeter with no data leaving Snowflake for AI inference.

API Endpoints

FastAPI is the single API server for all external consumers. It receives requests from the Slack Bot, Web Dashboard, and Streamlit test harness. It routes each request to the appropriate LangGraph agent, waits for completion, and returns the response.
Method Path Consumer Description
GET /health All Liveness and readiness check. Returns 200 if FastAPI is running and Snowflake connection is alive.
GET /deal/{id}/nba Dashboard, Slack Bot Returns ranked next best actions. Serves cached results if recent; otherwise triggers NBA Agent synchronously.
GET /deal/{id}/playbook Dashboard, Slack Bot Returns deal-specific playbook (11 sections). Serves cached results or triggers Playbook Agent.
GET /deal/{id}/prep Dashboard, Slack Bot Returns meeting prep brief. Combines deal context, Gong call summary, and open questions.
GET /deal/{id}/health Dashboard, Slack Bot Returns current health score breakdown and trend data. Pure read from the Semantic layer.
GET /pipeline/overview Dashboard Returns all open deals with health scores, risk levels, and summary metrics. Supports query params for filtering.
POST /feedback Dashboard, Slack Bot Records user feedback (thumbs up/down, rating, comment) on an NBA or playbook.
GET /agent/status Dashboard (admin), Streamlit Returns health and run history of each agent. Last run time, success/failure counts, average latency.
GET /user/preferences Dashboard, Slack Bot Returns the current user's notification preferences from the Hybrid Table.
Pure Reads (health, pipeline, stakeholders)
Latency: 50 to 150 ms
Reads directly from the Semantic layer or Hybrid Tables. No agent invocation. Timeout set to 5 seconds. This covers endpoints like /deal/{id}/health, /pipeline/overview, and cached NBA/playbook results.
Fresh Agent Call (nba/generate, playbook)
Latency: 2 to 15 seconds
Triggers a full LangGraph state machine execution including Cortex LLM inference. NBA generation takes 2 to 5 seconds. Playbook generation takes 5 to 15 seconds. Timeout set to 30 to 60 seconds respectively.

Event Hubs Topology

Azure Event Hubs provides real-time pub/sub between agents and between agents and delivery channels. When the Detection Agent finds events that need processing, it publishes them to Event Hubs. NBA and Playbook Agents consume from the same topic. Notifications flow back through a dedicated topic to the Slack Bot.
Topic Name Partitions Publisher Consumer(s)
fos-agent-events 4 Detection Agent (SPCS) NBA Agent (SPCS), Playbook Agent (SPCS)
fos-nba-results 2 NBA Agent (SPCS) Notification router
fos-playbook-results 2 Playbook Agent (SPCS) Notification router
fos-notifications 4 Notification router (SPCS) Slack Bot (Azure), Web Dashboard (via SSE or polling)
Decided: Partition key is hash(opportunity_id). All events for the same deal go to the same partition, enabling ordered processing per deal with no race conditions. The fos-notifications topic partitions by hash(target_user_id) to preserve per-user delivery order.
Retention
24 hours. Events older than 24 hours are stale. If an agent was down for a day, replaying old events would generate outdated recommendations. Short retention keeps the system forward-looking.
Dead Letter
3 attempts, then DLQ. Messages that fail processing 3 times are moved to a dead letter topic (fos-dlq) for manual inspection. Prevents poison messages from blocking the queue.
Checkpointing
Per consumer group, per partition. Consumers checkpoint after successful processing so they do not reprocess messages on restart. Each consumer group tracks its own offset independently.

Request Flows

Three detailed flows showing every hop in the system: a Slack command, a dashboard page load, and a proactive notification. Each flow traces from the user action (or system trigger) through every component to the final response.
Flow A: Slack Command 1-8 sec
User types a slash command like /nba Acme Corp in Slack.
  1. AE types slash command in Slack
  2. Slack POSTs webhook to Slack Bot on Azure
  3. Bot validates HMAC-SHA256 signature
  4. Bot returns immediate ack ("Looking up...")
  5. Bot resolves deal name to opportunity_id via SPCS
  6. Bot calls GET /deal/{id}/nba on SPCS
  7. FastAPI returns cached results (50 ms) or invokes NBA Agent (2-5 sec)
  8. Bot formats Block Kit message and posts DM

Cache hit: 1-2 sec. Cache miss: 4-8 sec (dominated by LLM call).
Flow B: Dashboard Load 120-150 ms
Manager clicks a deal in the pipeline view to open the deal detail page.
  1. Manager clicks deal name in browser
  2. Dashboard fires 5 parallel API requests to SPCS
  3. SPCS ingress terminates TLS, forwards to FastAPI
  4. FastAPI validates JWT, extracts user identity
  5. /health returns in 80 ms
  6. /nba returns cached results in 50 ms
  7. /playbook returns in 60 ms
  8. Dashboard renders progressively as responses arrive

Page interactive after slowest response: 120-150 ms. Refresh NBAs adds 2-5 sec for fresh LLM call.
Flow C: Proactive Alert 5-7 sec after detection
System detects a stale deal and proactively notifies the AE.
  1. Snowflake Task fires Detection Agent (every 60 min)
  2. Agent scans semantic views, finds stale deal
  3. Agent writes event to Hybrid Table
  4. Snowflake Stream detects new row
  5. Task publishes event to Event Hubs via Azure Function
  6. NBA Agent consumes event, runs LangGraph (2-5 sec)
  7. Agent publishes notification to fos-notifications
  8. Slack Bot consumes, formats Block Kit, sends DM

Detection to DM: 5-7 sec processing after the 60-minute polling interval.

Authentication

Authentication has the most open questions in this layer. MVP uses open visibility (any user can query any deal), but callers still need to be authenticated so the system knows who is making the request. Two integration paths need auth patterns: Slack Bot to SPCS and Dashboard to SPCS.
Slack Bot to SPCS TBD
The Slack Bot on Azure needs to prove its identity when calling SPCS FastAPI. Three options under evaluation:

Option 1: Snowflake OAuth (service-to-service)
Bot obtains OAuth token from Snowflake using client credentials grant. Snowflake-native. Token rotation is automatic. Requires OAuth integration setup.

Option 2: API Key (shared secret)
Long-lived key stored in the Bot's environment. Simple to implement but key rotation is manual. If leaked, full access until rotated.

Option 3: Key Pair RSA
RSA key pair authenticates as a Snowflake service user. JWT generated from the key pair. Strong authentication but more complex setup with Azure Key Vault.

Leaning toward: Snowflake OAuth (most Snowflake-native option).
Dashboard to SPCS TBD
Users access the Web Dashboard through a browser. The dashboard authenticates the user and passes their identity to SPCS. Three options under evaluation:

Option 1: SSO via Canva IdP + JWT
Dashboard authenticates against Canva's existing identity provider (Okta or Azure AD). IdP token exchanged for a short-lived JWT. Users log in with corporate credentials.

Option 2: Snowflake OAuth
User logs into Snowflake via browser redirect. Snowflake-native but requires every user to have a Snowflake login, which may not be desirable for AEs.

Option 3: Custom Auth
A lightweight auth service validates the user via IdP and issues a JWT with user metadata (role, team, Salesforce user ID). Flexible but adds another service to build.

Leaning toward: SSO via Canva IdP with JWT token exchange.
MVP Simplification: The system uses open visibility for MVP. No row-level security on deals. Any authenticated user can see any deal. This matches Canva's current Salesforce configuration. Auth only needs to (1) verify the caller is a legitimate FOS user, and (2) identify which user it is for feedback, preferences, and audit. Full RBAC (restricting AEs to their own deals, managers to their team) is a post-MVP enhancement.

SF-to-Event-Hubs Bridge

The Detection Agent runs as a Snowflake Task. After it writes events to a Hybrid Table, a chain of six components gets those events into Azure Event Hubs for real-time consumption by downstream agents. This is the most complex integration path in the application layer.
Hybrid Table fos_agent_events
→
SF Stream Detects new rows
→
SF Task STREAM_HAS_DATA
→
External Func publish_event_hubs
→
Azure Function HTTP relay
→
Event Hubs fos-agent-events
Six hops: Hybrid Table → SF Stream → SF Task → External Function → Azure Function → Event Hubs
Risk: This is the most fragile chain in the application layer. Six hops from Hybrid Table write to Event Hubs delivery. Each hop is a potential failure point. The SF Stream-to-Task trigger depends on SYSTEM$STREAM_HAS_DATA polling. The External Function-to-Azure Function call crosses the Snowflake-Azure boundary over HTTPS. This chain needs dedicated monitoring, alerting on each hop, and a clear escalation runbook.

Slack Bot to SPCS and Dashboard to SPCS		 Undecided Blocks Slack Bot integration and dashboard development. Must decide before either can call SPCS endpoints.
