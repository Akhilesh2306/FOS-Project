Ingestion Paths

There are three paths into the system. Each has a different source, connector, target, and level of readiness. Salesforce and Gong data land in Bronze as relational tables. Guru content does NOT go to Bronze. It gets chunked, embedded, and indexed directly into Cortex Search for vector similarity retrieval (RAG).
Salesforce Decided
Connector: Snowflake OpenFlow (Managed Apache NiFi, Bulk API)
Target: FOS_PROD.BRONZE
Frequency: Every 15 min
Data Type: 9 structured objects (CRM data)
Load Strategy: Incremental (CDC)

OpenFlow is Snowflake's native data integration service. Under the hood, it is a managed Apache NiFi instance. The pre-built Salesforce connector uses the Salesforce Bulk API for high-volume extraction with incremental sync (CDC), pulling only records that changed since the last sync.
Gong TBD
Connector: TBD (HTTP/API custom connector, Fivetran, or Gong Data Cloud)
Target: FOS_PROD.BRONZE
Frequency: TBD
Data Type: 16 structured objects (call data, transcripts, intelligence)
Load Strategy: TBD

OpenFlow does not have a Gong connector today. Three options are on the table: HTTP/API custom connector via OpenFlow, Fivetran's existing Gong connector, or Gong Data Cloud (zero-copy data sharing into Snowflake).
Guru / Knowledge Base TBD
Connector: TBD (Guru API, manual export, or batch loader)
Target: Cortex Search index (NOT Bronze)
Frequency: On content update
Data Type: Battle cards, case studies, win/loss reports, competitive intel
Load Strategy: Full re-index on change

Guru content is unstructured text meant for vector similarity search (RAG). Content goes through: Extract, Chunk, Embed (via Snowflake's embedding model), and Index into Cortex Search for fast similarity retrieval.

Latency Timeline

For Salesforce data, the worst-case latency chain from a field change by an AE to agent detection. Real-time triggers via Azure Event Hubs can reduce this for high-priority events, but the baseline polling path has this latency profile.
Source Change AE updates field in Salesforce
→

→

→
Gold 1 to 5 min
→
Semantic On query (zero latency)
→
Agent Detection Up to 60 min
Total worst case: ~75 to 90 minutes from Salesforce change to agent detection.
OpenFlow Sync
OpenFlow syncs every 15 minutes using incremental CDC via SystemModstamp. Only records changed since the last sync are pulled via the Salesforce Bulk API.
Dynamic Tables (Silver + Gold)
Silver and Gold layers use Snowflake Dynamic Tables that auto-refresh on upstream change. Refresh takes minutes, cascading from Silver to Gold.
Semantic + Agent
Semantic layer uses SQL Views (always current, no materialization, zero additional latency). Detection Agent scans every 60 minutes via Snowflake Task.

MEDDPICC Validation

Known Risk: MEDDPICC Custom Field Validation

The Opportunity object in Salesforce includes MEDDPICC custom fields (using the __c suffix convention). These fields are critical for the health scoring formula (15% weight in the AHP composite score). The 8 MEDDPICC components are: Metrics, Economic Buyer, Decision Criteria, Decision Process, Identify Pain, Champion, Competition, Paper Process. Each is scored 0 to 3.
What needs validation:
  • Confirm the exact API names of MEDDPICC custom fields in Canva's Salesforce org (e.g., meddpicc_metrics__c vs MEDDPICC_Metrics__c vs something else entirely)
  • Confirm the field data types (picklist with values 0 to 3? Number? Text?)
  • Confirm that these fields actually exist and are populated. If Canva uses a different field naming convention or a managed package for MEDDPICC, the Bronze DDL will need to be updated
Impact if wrong: Health scores will be incomplete. MEDDPICC gap detection will not work. Coaching nudges will be inaccurate.

Decisions

Summary of what has been decided, what remains open, and what carries risk for the data extraction pipeline.
Decided
  • Salesforce extraction uses OpenFlow with the Salesforce Bulk API connector
  • Salesforce sync runs every 15 minutes, incremental
  • 9 Salesforce objects have been identified and DDL is defined
  • 16 Gong objects have been identified and DDL is defined
  • Guru/content data goes to Cortex Search, not to Bronze
  • All structured data lands in FOS_PROD.BRONZE with no transformations
  • Bronze has 35 total tables (25 external + 10 Hybrid Tables)
  • FOS backend uses Hybrid Tables with direct SPCS agent writes
  • Dynamic Tables handle Bronze-to-Silver and Silver-to-Gold refresh automatically
  • Semantic layer uses SQL Views (always current, not materialized)
  • The architecture is Snowflake-native. No dbt. No external orchestration for transformations.
  • Canva currently uses Fivetran for some SFDC sync, but FOS uses its own OpenFlow pipeline independently
  • Table naming convention is lowercase (intentional, not Fivetran defaults)
Open Questions
  • Which Gong connector to use (HTTP/API via OpenFlow, Fivetran, or Gong Data Cloud)?
  • What is the Gong sync frequency?
  • Do Gong transcripts also go to Cortex Search for RAG?
  • How do we extract content from Guru? (Guru REST API, bulk export, or batch loader)
  • What is the chunking and embedding strategy for Cortex Search?
  • What is the exact inventory of content in Guru?
  • What are the exact API names of MEDDPICC custom fields in Canva's SFDC org?
  • Does OpenFlow support the specific Salesforce custom fields Canva uses?
  • What is the Gong API rate limit?
  • Does Canva have a Gong Data Cloud license?
  • How do we handle Guru content updates and re-indexing?
Risks
  • Gong connector blocks 16 Bronze tables. No Gong connector means no call analytics, no competitor detection, no SE handoff signals, and no meeting prep. This is the single largest blocker in the extraction pipeline.
  • Wrong MEDDPICC field names cause silent failures. If the custom field API names do not match what the Bronze DDL expects, MEDDPICC scoring (15% of the health score) breaks silently. Health scores will be incomplete, gap detection will not work, and coaching nudges will be inaccurate.
  • Guru extraction is fully undefined. Almost everything about this path is undefined: extraction method, chunking strategy, embedding model, update handling, content inventory, metadata preservation, and access controls. Without Guru content, RAG returns nothing for battle cards and win/loss reports.
