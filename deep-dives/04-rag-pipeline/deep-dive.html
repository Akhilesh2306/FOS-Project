RAG and Content Pipeline

Separate from medallion. Unstructured knowledge from Guru to Cortex Search. Two-stage retrieval: SQL filter then vector similarity.

1

Key Distinction

The RAG pipeline is a completely separate system from the medallion data architecture. They serve different purposes, handle different data types, and follow different paths. Understanding this separation is critical to understanding the FOS architecture.
DECIDED: Medallion = structured data (SFDC, Gong) through Dynamic Tables. RAG = unstructured knowledge (Guru) through Cortex Search. Two separate pipelines.
Aspect Medallion Architecture RAG Content Pipeline
Data type Structured (Salesforce, Gong) Unstructured (documents, cards, reports)
Source SFDC via OpenFlow, Gong via connector Guru knowledge base (primary)
Destination Bronze, Silver, Gold, Semantic Cortex Search index
Consumed by All agents (via Semantic views) RAG Agent (serves NBA and Playbook agents)
Storage format Relational tables and Dynamic Tables Chunked text with vector embeddings
Query method Against Semantic views Two-stage: metadata filter then vector similarity
Relationship to agents: The RAG Agent is not a standalone consumer. It is a service agent called by the NBA Agent and Playbook Agent when they need knowledge content to enrich their outputs. When the NBA Agent generates recommendations for a stale deal involving a competitor, it calls the RAG Agent to retrieve relevant battle card sections. When the Playbook Agent builds a deal playbook, it calls the RAG Agent for objection handling scripts and case studies.

Extraction Options

Four approaches for extracting content from Guru. This is the most uncertain part of the pipeline. The right approach depends on Canva's Guru instance configuration, content volume, and update patterns.
Option A: REST API (Card-by-Card) LIKELY PROD
Schedule a Snowflake Task or cron job. Call Guru API to list cards modified since last sync. Pull full content per card. Write to staging table.

Pros: Incremental sync (only pull changes). Fine-grained control. Can pull individual card metadata and tags.

Cons: Card-by-card calls could be slow for large collections. No published rate limits. Requires building and maintaining API integration. HTML content needs parsing.
Option B: Collection Export (Bulk) TBD
POST to export endpoint with webhook callback. Download ZIP on completion. Parse card files and diff against previous export.

Pros: Gets everything in one operation. Includes folder structure and tags. Simpler than many individual API calls.

Cons: Full export every time (no incremental). Requires diffing to detect changes. Async workflow with webhook adds complexity.
Option C: Manual Upload MVP START
Content team exports from Guru UI manually. Uploads files to S3 bucket or Snowflake internal stage. Snowflake Task detects and ingests new files.

Pros: No API integration needed. Content team controls what gets indexed. Simplest to implement for MVP.

Cons: Manual process, does not scale. Content can go stale between uploads. No automation. Relies on human discipline.
Option D: Webhooks (Real-Time) UNCERTAIN
Register a webhook endpoint. Guru fires webhook on content events. Handler calls Guru API to pull updated card. Writes to staging and triggers re-processing.

Pros: Near real-time content freshness. Event-driven (no polling). Only processes actual changes.

Cons: Documented webhook events seem limited (card-copied, card-restored, link views). Not clear if card-updated or card-created events exist. Requires always-on endpoint.
Recommended MVP approach: Start with Option C (manual upload) in Sprint 1 to unblock downstream pipeline development. In parallel, investigate Options A and D to determine the right automated approach. Option A (REST API with incremental sync) is the most likely production solution.

Chunking Strategy

Chunking is how we split source documents into segments for embedding and retrieval. This is a critical decision because it directly impacts retrieval quality. Bad chunks produce bad search results, which produce bad agent outputs.
Key constraint: Snowflake Cortex Search embeds only the first 512 tokens of each text entry for vector search. Content beyond 512 tokens is still indexed for keyword search but will not influence semantic similarity matching. This constraint anchors the chunking strategy.
Aspect Heading-Based RECOMMENDED Fixed-Size (512 tokens) FALLBACK
How it works Split documents at section boundaries (H1, H2, H3). Each section becomes one chunk. If a section exceeds 512 tokens, split further with overlap. Split every document into sequential chunks of exactly 512 tokens with configurable overlap (typically 50-100 tokens).
Simplicity Medium. Requires heading detection. High. Easy to implement, deterministic.
Battle Cards Excellent. Clear headings: Overview, Strengths, Weaknesses, Pricing, Objection Handling. Poor. Sections get split mid-thought. A "Pricing Objections" section might span two chunks.
Case Studies Good. Natural sections: Background, Challenge, Solution, Results. Mediocre. Narrative flow gets broken arbitrarily.
Win/Loss Reports Good. Structured with Key Findings, What Worked, What We Lost On. Mediocre. Key findings may split across chunks.
Retrieval Precision Higher. Each chunk is topically coherent. Lower. Chunks may contain mixed topics.
Requirement Source content must have consistent heading structure. No structural requirement on source content.
Recommended approach: Use heading-based chunking as the primary strategy. When a section exceeds 512 tokens, apply fixed-size splitting with 100-token overlap within that section. For documents without headings (e.g., competitive intel research notes), fall back to fixed-size chunking. This hybrid approach handles both structured and unstructured content.
Content Type Recommended Strategy Rationale
Battle Cards Heading-based Highly structured, clear sections (one chunk per section)
Win/Loss Reports Heading-based with overflow splitting Structured but some sections run long
Case Studies Heading-based with overflow splitting Narrative sections may exceed 512 tokens
Product Docs Heading-based Technical docs are well-structured
Sales Methodology Heading-based with overflow splitting Mixed structure: short tips and long explanations
Competitive Intel Fixed-size with overlap Often unstructured research notes

Two-Stage Retrieval

The core retrieval pattern used by the RAG Agent. First narrow the search space with structured metadata filters (fast), then run vector similarity on the reduced set (precise). Faster and more accurate than a naive full-corpus vector scan.
Stage 1: Metadata Filter DECIDED
The RAG Agent receives context from the calling agent: industry, deal stage, products, detected competitors, and request type (battle card, case study, objection handling).

Using this context, the agent constructs a metadata filter:

content_type IN (Battle Card, Win Report)
competitors contains "Figma"
industries contains "Technology"
deal_stages contains "Negotiation"

This typically reduces the search space from thousands of chunks to 20-50 relevant chunks.

Performance target: under 100ms
Stage 2: Vector Similarity DECIDED
With the narrowed set, run vector similarity using the actual query text (e.g., "How should we position against Figma enterprise pricing in negotiation?").

Cortex Search handles this with hybrid search: vector retrieval (cosine similarity), keyword retrieval (BM25-style), and semantic reranking. Hybrid produces 12%+ better accuracy than pure vector search alone.

Returns top 5-10 most relevant chunks with: chunk text, source document title, source URL (link back to Guru), content type, relevance score, section title (for citation).

Performance target: under 400ms

Needs validation: Cortex Search ATTRIBUTES filtering may be sufficient to replace the explicit two-stage pattern. When you query with a filter, the filter is applied before the search, meaning Stage 1 and Stage 2 could be a single API call. If built-in filtering is performant enough, we can simplify the RAG Agent. If not, we keep the explicit two-stage approach with v_rag_content_catalog for Stage 1 and Cortex Search for Stage 2.

Open Questions

The RAG pipeline is the least defined part of the FOS architecture. Most decisions are preliminary or TBD. This section captures the unknowns that must be resolved before production.
17 unknowns identified. This is the LEAST defined part of the FOS architecture. Key blockers that must be resolved before production:
  • Guru API access -- Need credentials from Canva IT. Extraction development is blocked without this.
  • Guru instance audit -- Unknown: card count, collection structure, tagging quality, content formats. Cannot size infrastructure or plan metadata enrichment.
  • Chunking validation -- Need 10-20 sample Guru cards to test heading-based chunking. Do battle cards have consistent heading structures?
  • Cortex Search ATTRIBUTES behavior -- Does built-in filtering replace explicit two-stage retrieval? Determines RAG Agent complexity.
