Tool Inventory

Every tool is a Python function imported and called directly inside LangGraph nodes. Tools interact with Snowflake, Cortex, and Azure Event Hubs on behalf of agents.
Decided: All tools are plain Python functions. No LangChain tool wrappers, no schema decorators. Each function takes explicit parameters and returns results. This keeps the codebase simple and avoids unnecessary abstraction.
Tool What It Does Used By
query_semantic_view Executes a query against a Semantic layer view via Snowpark SDK. Returns rows as a list of dicts. Detection, NBA, Playbook, RAG
call_cortex_llm Calls Snowflake Cortex LLM for text generation. Takes prompt, temperature, max_tokens. Returns generated text. NBA, Playbook
search_cortex Calls Cortex Search for vector similarity retrieval. Takes query string and optional metadata filters. Returns ranked content chunks. RAG (called by NBA, Playbook)
write_hybrid_table Inserts or updates a row in a Hybrid Table via Snowpark SDK. Takes table name, column dict, operation type. Detection, NBA, Playbook, Feedback
publish_event_hubs Publishes a JSON message to an Azure Event Hubs topic. Returns delivery confirmation. Detection, NBA, Playbook
consume_event_hubs Receives messages from an Event Hubs consumer group. Takes topic name and batch size. Returns list of event messages. NBA, Playbook
log_agent_run Writes execution metadata to fos_agent_runs: agent name, timestamps, status, error details, node trace. All agents

Execution Pattern

Every agent follows the same graph execution pattern. The graph is compiled once at startup. Each invocation creates a new state, executes nodes in sequence, and checkpoints after every step.
Receive Request HTTP or Event Hubs
→
Fetch Context Semantic views
→
Check History Cooldown list
→
Generate Cortex LLM
→
Rank / Filter In-memory scoring
→
Write Results Hybrid Tables
→
Publish Event Event Hubs
Decided: Checkpoint saved after each step via langgraph-checkpoint-snowflake. If any node fails, the agent resumes from the last successful checkpoint without re-executing earlier steps. The most expensive step (LLM call) is never unnecessarily repeated.
State lifecycle: PENDING -> FETCHING_CONTEXT -> GENERATING -> RANKING -> WRITING -> COMPLETE (or FAILED at any step)

Error Handling

All tools follow the same error handling convention: transient failures raise RetryableError (retried with backoff), permanent failures raise PermanentError (logged and skipped or marked FAILED).
Retry Pattern
Max retries: 3
Backoff: Exponential: 2s, 4s, 8s
Jitter: Random 0-1s added to each backoff
Scope: Node-level, not graph-level. Only the failing node is retried, not the entire graph.
LLM Timeout
Timeout: 30 seconds per Cortex call
Recovery: Retry up to 3x with exponential backoff
If all retries fail: Mark event as FAILED in fos_agent_events. Checkpoint preserved at pre-LLM node for manual retry.
Cortex Search Down
Impact: No RAG content available for NBA or Playbook generation
Recovery: Skip RAG content entirely. Proceed with LLM generation using deal context only.
Result: Functional but degraded. Recommendations lack battle card and competitive intel context.
Hybrid Table Write Fails
Impact: Recommendations generated but not persisted
Recovery: Resume from checkpoint. State contains the generated results. Retry skips LLM regeneration and goes straight to write.
Transactions: ACID rollback on partial writes.

State Management

Agent state is persisted to Snowflake after every graph node using the langgraph-checkpoint-snowflake package. This replaces the default PostgreSQL or SQLite checkpoint backends with native Snowflake storage.
Decided: langgraph-checkpoint-snowflake for all state persistence. Each checkpoint is keyed by thread_id (unique per agent run, format: {agent}_{opp_id}_{ts}) + checkpoint_ts (timestamp). State is serialized to JSON and stored as a VARIANT column. No PostgreSQL anywhere in the stack.
Policy Retention Detail
Active checkpoints 7 days Supports conversation continuity for Slack threads and debugging of in-progress runs
Completed runs 30 days Final checkpoint retained for audit. Intermediate checkpoints pruned after 7 days.
Failed runs 14 days Retained for investigation. Includes full state at failure point and error details.
Cleanup mechanism Daily Snowflake Task runs daily to delete expired checkpoints based on TTL policies
Estimated volume: Each checkpoint is 10-200 KB. At 500 agent runs/day with 5 checkpoints each, approximately 250-500 MB/month before cleanup.

Open Questions

Several architecture decisions remain unresolved. These items block downstream work including prompt engineering, capacity planning, and integration testing.
Open: Cortex LLM model selection (Claude vs Mistral Large vs Llama 3) is not finalized. This blocks prompt engineering, cost estimation, and output quality benchmarking. The architecture supports switching models without code changes (model name is a config parameter), but prompt templates may need tuning per model.
SPCS Scaling Limits TBD
Need to confirm max replicas, auto-scale trigger mechanisms, and cold start behavior for SPCS compute pools. Documentation is sparse. Affects capacity planning and SLA commitments.
Cortex Rate Limits TBD
Need to understand tokens-per-minute and requests-per-minute limits for the chosen model. Affects how many deals can be processed per detection cycle. Blocks capacity planning.
SPCS Outbound Networking TBD
SPCS may restrict outbound HTTPS access. Need to verify external access list configuration for Azure Event Hubs and LangSmith connectivity. Blocks Event Hubs integration.
Cortex Search Index Limits TBD
Need to understand how much RAG content (battle cards, win/loss reports) can be indexed. Affects content ingestion pipeline design. Content volume is small for MVP.
