Semantic Layer

11 SQL views that agents query. The only data interface for AI. Agents never touch Gold, Silver, or Bronze.

1

Architecture Rule

DECIDED: Agents query ONLY the Semantic layer. Never Gold, Silver, or Bronze.

This is the single most important data access rule in the FOS system. If an agent needs data that does not exist in a Semantic view today, the correct response is to add a column to an existing view or create a new Semantic view. The incorrect response is to let the agent query Gold.

Clean Interface Contract
The Semantic layer is a stable API between two teams: data engineering (who owns Bronze through Gold) and AI engineering (who builds agents in SPCS). Each team works independently as long as Semantic view signatures stay stable.
Decouples Agents from Transform Logic
Gold views contain complex CTEs, weighted scoring formulas, window functions, and aggregation logic. With Semantic as the boundary, Gold logic can evolve freely. The Semantic view absorbs the change and continues to expose the same column names and types.
Medallion Changes Without Breaking Agents
The project may refactor how Silver computes days_since_last_activity or how Gold weights health score components. Those changes cascade through Dynamic Table auto-refresh and land in Semantic views transparently. Agents never know or care.

Naming Convention

Semantic layer columns use business-friendly English names. The rule: if you put the column name in an LLM prompt, a non-technical person should understand what it means without a data dictionary.
Instead of Use this Rule
opp_name__c opportunity_name Readable in prompts, no Salesforce suffixes
acct_industry account_industry No abbreviations
calc_health_score_v2 health_score No internal versioning exposed
meddicc_eb_score economic_buyer_score Spell out acronyms where possible
days_since_last_act days_since_last_activity No truncation
is_stale_flag is_stale No redundant suffixes
amt amount Full words always
opp_close_dt close_date Standard date naming
Why this matters for LLMs. When an agent builds a prompt like "The deal {opportunity_name} has a health score of {health_score} and the economic buyer has not been identified," every column name becomes part of the natural language context. Cryptic names like calc_hs_v2 force the agent to include a translation layer in its prompt template, adding token count and reducing clarity. Clean names also help Snowflake Cortex Analyst, which introspects view schemas and uses column names as semantic hints when generating SQL from natural language.

Naming Rules

snake_case for all column names
Full English words, no abbreviations except universally understood ones (id, url, pct)
Booleans start with is_ or has_
Scores end with _score
Counts end with _count
Percentages end with _pct
Dates end with _date or _at
Summaries end with _summary or _description

Performance

Why SQL Views
Always current. No refresh lag. When the agent queries, it gets the latest Gold data at that moment.

Simpler to manage. No refresh schedules, no compute cost for materialization.

Gold handles the heavy transforms. Semantic views are thin wrappers. The complex CTEs, scoring formulas, and window functions live in Gold Dynamic Tables, which refresh automatically.
Trade-off
Complex views execute full query plan on every call. The key view v_opportunity_llm_context uses correlated subqueries for stakeholder_map, recent_activities_summary, and recent_gong_calls_summary. These are evaluated per-row.

Snowflake result cache helps. If the same query runs twice within 24 hours and underlying data has not changed, the second execution returns cached results in milliseconds.

Pipeline-level scans are heavier. v_pipeline_overview and v_agent_performance scan all open opportunities, but they are less latency-sensitive (dashboard and batch use cases).
OPEN: Fallback plan if 500ms is not achievable. If plain views cannot meet the 500ms target at production data volumes, the fallback is to convert v_opportunity_llm_context into a Dynamic Table with a short refresh interval (e.g., 1 minute). This trades absolute freshness for guaranteed performance. The decision should be data-driven based on actual query latency measurements during the pilot.

Optimization Strategies

1. Ensure clustering/indexes on opportunity_id across all joined tables

2. Limit subqueries to bounded sets (last 10 activities, last 3 Gong calls)
3. Use STRING_AGG for efficient single-pass concatenation

4. If subquery performance degrades at scale, pre-compute aggregations into Gold Dynamic Tables

Decisions

DECIDED
  • 11 views in FOS_PROD.SEMANTIC for MVP
  • SQL views, not Dynamic Tables or materialized views
  • Business-friendly column names. No Salesforce __c suffixes in Semantic
  • Agents query Semantic only. Never Gold, Silver, or Bronze

OPEN
  • MEDDPICC vs MEDDICC scoring total. Some parts score out of 21 (7 components), others out of 24 (8 with Paper Process). Needs one canonical total.
  • 6 views still need SQL written. Only v_opportunity_llm_context has a full SQL definition. The other Semantic wrapper views have not been authored yet.
  • Naming gap. PRD uses names like v_deal_health_summary. Storybook uses v_opportunity_health_current. Gold uses its own names. Canonical mapping needed.
  • Gong data fallback. If Gong connector is not available at launch, Gong columns return NULLs. Should views return "No Gong data available" strings instead?
  • Actual latency at scale. The 500ms target for v_opportunity_llm_context is an assumption. Not benchmarked against real opportunity volumes.
  • v_pipeline_overview aggregation level. Per-rep, per-team, or per-company? PRD mentions manager pipeline reviews (team-level), but the view is not defined yet.
